The Core Misunderstanding (Very Common)
Many people think:
“Tight coupling exists only when I have multiple implementations (Airtel, Jio, Idea).
If I have only Airtel, why bother with DI?”
This sounds logical — but it’s incomplete.
________________________________________
What Tight Coupling Actually Means
Tight coupling is NOT about number of classes.
It is about who controls object creation.
Tight Coupling =
“This class decides what concrete dependency it will use.”
Loose Coupling =
“This class does NOT decide. Someone else provides the dependency.”
Even one single class can still be tightly coupled.
________________________________________
Single-Class Example (Very Important)
Case 1: Without DI (Tight Coupling)
class Mobile {
    private AirtelSim sim = new AirtelSim();
}
You might say:
“There’s only Airtel. What’s the problem?”
The hidden problems:
1.	Mobile is tied to AirtelSim class
2.	You cannot test Mobile without AirtelSim
3.	You cannot replace AirtelSim without modifying Mobile
4.	Mobile controls creation → violation of SRP
5.	Any future change forces code change
This is tight coupling even with one class.
________________________________________
Why This Is Still a Problem in Real Projects
Think like a real enterprise system:
Today:
•	Only AirtelSim exists
Tomorrow:
•	AirtelSim needs DB
•	AirtelSim needs API call
•	AirtelSim needs caching
•	AirtelSim needs mock for testing
•	AirtelSim becomes heavy
Now what?
You must:
•	Change Mobile code
•	Recompile
•	Retest
•	Redeploy
That’s fragile architecture.
________________________________________
Same Scenario With DI (Loose Coupling)
class Mobile {
    private Sim sim;

    Mobile(Sim sim) {
        this.sim = sim;
    }
}
Now:
•	Mobile does not know AirtelSim exists
•	Mobile depends on abstraction
•	Creation happens outside
Even if there is only one implementation today, you gain:
✔ Testability
✔ Replaceability
✔ Flexibility
✔ Clean architecture
✔ SRP compliance
________________________________________
Real-Life Analogy (Very Clear)
Tight Coupling (Even With One Option)
You buy a mobile where:
•	The SIM is soldered inside
•	Even if only Airtel exists today
Still a bad design.
Why?
Because:
•	You cannot replace it
•	You cannot remove it
•	You cannot test without it
•	You cannot evolve it
________________________________________
Loose Coupling (DI)
You buy a phone with a SIM slot.
Even if:
•	Only Airtel SIM exists today
The design is still correct.
Because:
•	The phone is not responsible for SIM creation
•	The phone accepts SIM from outside
•	The phone is future-proof
________________________________________
Why Spring Enforces DI Even With One Class
Spring is not solving today’s problem.
Spring is solving tomorrow’s change.
Spring assumes:
•	Requirements WILL change
•	Testing WILL be needed
•	Implementations WILL grow
•	Production issues WILL occur
So Spring enforces:
•	Constructor injection
•	Interface-driven design
•	External wiring
Not for convenience — but for survivability.
________________________________________
Interview-Grade One-Liner (Memorize This)
“Tight coupling is not about how many implementations exist.
It’s about who controls object creation.
DI removes that control from the class, even if there’s only one implementation.”

Yes — even if there is only ONE implementation today, Spring still encourages DI.
Not because you need choices now, but because tight coupling is about control, not count.
Now let’s deeply understand why.
________________________________________


When Is new Actually Acceptable?
Good question.
You can use new safely when:
•	Class is immutable
•	No external dependency
•	No business behavior
•	Utility / value objects
Example:
new BigDecimal("100");
new LocalDate();
But not for services, repositories, integrations, or business logic.
